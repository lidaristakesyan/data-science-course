MapReduce

The foundational big data processing model
MapReduce is a programming model for processing huge datasets across many machines with two phases: MAP (transform each record independently) and REDUCE (combine/aggregate results).
Word Count Example
INPUT (3 machines):
Machine 1: "hello world"
Machine 2: "hello there"
Machine 3: "world world"

STEP 1: MAP (parallel)
M1: (hello,1), (world,1)
M2: (hello,1), (there,1)
M3: (world,1), (world,1)

STEP 2: SHUFFLE (group by key)
hello ‚Üí [(hello,1), (hello,1)]
world ‚Üí [(world,1), (world,1), (world,1)]
there ‚Üí [(there,1)]

STEP 3: REDUCE (aggregate)
hello: 1+1 = 2
world: 1+1+1 = 3
there: 1
Key Concepts for Data Scientists
  • Parallel Processing: Work divided across machines
  • Data Locality: Process data where it's stored
  • Fault Tolerance: Failed tasks automatically retry
  • Scalability: Add machines for more data
Limitation:
MapReduce writes to disk after each step ‚Äî slow for iterative algorithms (ML)
Time: O(n/p) + O(log p)
n=data, p=machines

Apache Spark

Fast, general-purpose distributed processing engine
‚ö° Why Spark is Faster
MapReduce (slow):
Step 1 ‚Üí Disk ‚Üí Step 2 ‚Üí Disk ‚Üí Step 3

Spark (fast):
Step 1 ‚Üí Memory ‚Üí Step 2 ‚Üí Memory ‚Üí Step 3

Result: Spark is 10-100x faster for iterative algorithms (ML)

  • In-memory processing
  • Lazy evaluation (optimized execution)
  • Rich API (Python, Scala, R, SQL)
  • Unified engine for batch + streaming
üêç PySpark Example
from pyspark.sql import SparkSession

# Initialize Spark
spark = SparkSession.builder.getOrCreate()

# Read distributed data
df = spark.read.csv("s3://data/sales/")

# Transformations (lazy)
result = df \
  .filter(df.amount > 100) \
  .groupBy("product") \
  .agg({"amount": "sum"})

# Action (triggers execution)
result.show()
Key: Transformations are lazy, Actions trigger computation
Use Pandas for <1GB, Spark for >1GB ‚Äî choose the simplest tool that handles your data size

OLTP vs OLAP

Transactional vs Analytical databases
‚öôÔ∏è OLTP (Transactional)

"Operations" database ‚Äî day-to-day business

  • Purpose: Process individual transactions
  • Examples: Add to cart, update profile, payments
  • Optimized for: WRITES (insert, update, delete)
  • Data volume: Single records, small batches
  • Users: Thousands concurrent
  • Storage: Row-oriented
Systems: PostgreSQL, MySQL, Oracle
üìä OLAP (Analytical)

"Analytics" database ‚Äî complex analysis

  • Purpose: Analyze historical data
  • Examples: Revenue trends, segmentation, forecasting
  • Optimized for: READS (complex queries)
  • Data volume: Millions to billions of records
  • Users: Fewer (analysts, data scientists)
  • Storage: Column-oriented
Systems: Snowflake, BigQuery, Redshift
Data Scientist Insight: You mostly query OLAP systems, but understanding OLTP helps you know where data originates.

Essential SQL for Data Scientists

The queries you'll use every day
Basic Queries
-- SELECT with filter
SELECT name, email
FROM users
WHERE city = 'New York';

-- Aggregation
SELECT department,
       COUNT(*) as count,
       AVG(salary) as avg_sal
FROM employees
GROUP BY department
HAVING AVG(salary) > 70000;

-- Sorting & Limiting
SELECT * FROM products
ORDER BY price DESC
LIMIT 10;
JOINs
-- INNER JOIN (only matches)
SELECT u.name, o.amount
FROM users u
INNER JOIN orders o
  ON u.user_id = o.user_id;

-- LEFT JOIN (all from left + matches)
SELECT u.name,
       COUNT(o.order_id) as orders
FROM users u
LEFT JOIN orders o
  ON u.user_id = o.user_id
GROUP BY u.user_id, u.name;

-- Multiple JOINs
SELECT u.name, p.product_name, o.amount
FROM orders o
JOIN users u ON o.user_id = u.user_id
JOIN products p ON o.product_id = p.id;
Master: SELECT, JOIN, WHERE, GROUP BY, HAVING, ORDER BY, LIMIT ‚Äî covers 90% of data science SQL
